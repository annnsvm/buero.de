# Вимоги до форми створення курсу та приклади

Документ описує вимоги до UI форми створення/редагування курсу, модулів і матеріалів, а також приклади структур даних і відображення. Призначений для фронтенду та узгодження з бекендом.

---

## 1. Поле `content` у БД — універсальне поле

У таблиці **course_materials** поле **content** має тип **JSON (JSONB)**. Воно не має фіксованої схеми для всіх типів матеріалів одночасно.

- **Різні типи матеріалів** (`video`, `vocabulary`, `quiz`, `scenario`, тощо) використовують **різну структуру** об'єкта в `content`.
- Допускається **різна логіка та реалізація** на фронтенді: як саме формувати JSON при збереженні і як його інтерпретувати при відображенні — визначається типом матеріалу та продуктовими вимогами.
- Форма створення/редагування матеріалу може мати **різні поля вводу** залежно від обраного типу; ці поля разом складаються в один об'єкт `content` при відправці на API.
- Таким чином, `content` — **універсальне поле** для зберігання контенту: конкретна схема та UX форм і вьюерів можуть еволюціонувати окремо для кожного типу матеріалу, без зміни типу колонки в БД.

Бекенд приймає `content` як об'єкт (валідація за типом — опційно, напр. для `scenario` мінімум 3 гілки та `feedback` у choices). Фронтенд сам вирішує, які поля показувати у формі і як рендерити матеріал на сторінці перегляду.

---

## 2. Ієрархія та етапи форми

- **Курс** → **Модулі** → **Матеріали** (всередині кожного модуля).
- Логічний порядок у формі:
  1. Створення або редагування курсу (поля курсу).
  2. Додавання/редагування модулів курсу (кожен модуль: назва, порядок).
  3. Для обраного модуля — додавання/редагування матеріалів (тип, назва, порядок + контент за типом).

Приклад потоків:

- **Створення курсу:** Заповнити курс → Зберегти → Додати модуль(и) → Обрати модуль → Додати матеріали.
- **Редагування:** Обрати курс → Редагувати курс / Модулі / Матеріали в межах модуля.

## 3. Вимоги до форми курсу (Course)

| Поле           | Обов'язкове | Тип / обмеження              | Поле БД              | Приклад відображення (студент)     |
|----------------|-------------|------------------------------|----------------------|------------------------------------|
| Назва курсу    | Так         | Текст, макс. 500 символів    | `courses.title`      | Заголовок картки/сторінки курсу    |
| Опис           | Ні          | Багаторядковий текст          | `courses.description` | Під заголовком, в блоці «Про курс» |
| Мова контенту  | Так         | Вибір: `en` \| `de`           | `courses.language`   | Мітка мови / фільтр каталогу       |
| Категорія      | Так         | Вибір: `language` \| `sociocultural` | `courses.category` | Розділ каталогу (напр. «Мова» / «Життя в Німеччині») |
| Опублікований  | Ні          | Чекбокс (за замовч. false)    | `courses.is_published` | Показ у каталозі / приховування    |

**Приклад заповненої форми (курс):**

- Назва: «Німецька A1: Основи»
- Опис: «Курс для початківців. Лексика, граматика, прості діалоги.»
- Мова: `en`
- Категорія: `language`
- Опублікований: ☐

## 4. Вимоги до форми модуля (Course Module)

| Поле          | Обов'язкове | Тип              | Поле БД                   | Приклад відображення        |
|---------------|-------------|-------------------|---------------------------|-----------------------------|
| Назва модуля  | Так         | Текст, макс. 500  | `course_modules.title`    | Заголовок модуля в сайдбарі |
| Порядок       | Так         | Ціле число ≥ 0    | `course_modules.order_index` | Порядок модулів у списку  |

**Приклад заповненої форми (модуль):**

- Назва: «Модуль 1: Знайомство»
- Порядок: `0`

## 5. Спільні поля матеріалу (усі типи)

Для кожного матеріалу (Course Material):

| Поле   | Обов'язкове | Тип            | Поле БД                    | Відображення                 |
|--------|-------------|----------------|----------------------------|------------------------------|
| Тип    | Так         | Вибір з enum типів | `course_materials.type`   | Іконка/мітка типу уроку      |
| Назва  | Так         | Текст, макс. 500 | `course_materials.title`  | Заголовок уроку в списку     |
| Порядок| Так         | Ціле число ≥ 0   | `course_materials.order_index` | Порядок уроків у модулі |

Після вибору типу матеріалу форма показує **додаткові поля для контенту**. Вони збираються в об'єкт і зберігаються в **content**. Нижче — рекомендовані структури та приклади; реалізація на фронті може відрізнятися (наприклад, інші ключі або додаткові поля), якщо це узгоджено з бекендом і відображенням.

## 6. Контент по типах матеріалу — приклади структур і форм

### 6.1. video

- **Призначення:** одне відео (MVP — YouTube).
- **Поля форми:** одне поле — «ID відео YouTube» (рядок, тільки id, напр. `dQw4w9WgXcQ`).
- **Приклад content у БД:**
```json
{
  "youtube_video_id": "dQw4w9WgXcQ"
}
```
- **Відображення:** плеєр (iframe або YouTube IFrame API) за `content.youtube_video_id`. Інша реалізація (напр. інший ключ або майбутній S3) — можлива за умови узгодженої схеми в `content`.

---

### 6.2. text

- **Призначення:** статичний текст (або HTML).
- **Поля форми:** одне поле — «Текст» (багаторядкове; опційно rich text).
- **Приклад content у БД:**
```json
{
  "body": "<p>Текст уроку. Можна використовувати HTML.</p>"
}
```
або простіший варіант:
```json
{
  "text": "Звичайний текст без HTML."
}
```
- **Відображення:** вивід `content.body` або `content.text` (з санітизацією HTML, якщо потрібно). Конкретний ключ і формат — на розсуд фронту, головне консистентно зберігати і читати.

---

### 6.3. vocabulary

- **Призначення:** список слів/фраз із перекладом або поясненням.
- **Поля форми:** список пар «слово — переклад/пояснення» (кнопки «Додати ряд», «Видалити ряд»).
- **Приклад content у БД:**
```json
{
  "items": [
    { "word": "das Buch", "translation": "книга" },
    { "word": "lesen", "translation": "читати" }
  ]
}
```
- **Відображення:** список або картки слів з `content.items` (наприклад, word + translation). Можна додати власні поля в кожен item (напр. `example`, `audio_url`) — тоді форма і вьюер налаштовуються під цю розширену схему.

---

### 6.4. grammar

- **Призначення:** правило граматики з описом і прикладами.
- **Поля форми:** заголовок правила (опційно), основний текст, приклади (список рядків або окремі поля).
- **Приклад content у БД:**
```json
{
  "title": "Артикль der, die, das",
  "body": "У німецькій три артиклі...",
  "examples": ["der Mann", "die Frau", "das Kind"]
}
```
- **Відображення:** блок «Граматика»: title, body, список examples. Структуру можна розширювати (напр. підзаголовки, таблиці в JSON).

---

### 6.5. cultural_insight

- **Призначення:** культурна вставка (текст + опційно посилання).
- **Поля форми:** заголовок (опційно), текст, URL та підпис посилання (опційно).
- **Приклад content у БД:**
```json
{
  "title": "Anmeldung",
  "body": "Реєстрація за місцем проживання в Німеччині...",
  "link": { "url": "https://example.com", "label": "Детальніше на сайті" }
}
```
- **Відображення:** блок з title, body і кнопкою/лінком за `content.link`. Поля в `content` можна змінювати під дизайн (напр. додати `image_url`).

---

### 6.6. homework

- **Призначення:** опис домашнього завдання.
- **Поля форми:** опис (обов'язково), дедлайн (текст або дата), критерії прийняття (опційно).
- **Приклад content у БД:**
```json
{
  "description": "Напишіть 5 речень про ваш день.",
  "deadline": "На наступний урок",
  "criteria": "Використати час Perfekt."
}
```
- **Відображення:** блок «Домашнє завдання»: description, deadline, criteria. Збереження виконання — окремо (Progress/Homework), не в `content`.

---

### 6.7. quiz

- **Призначення:** тест із питаннями та варіантами відповідей.
- **Поля форми:** блоки питань; для кожного питання: текст, варіанти відповідей, позначка правильної відповіді.
- **Приклад content у БД:**
```json
{
  "blocks": [
    {
      "questions": [
        {
          "id": "q1",
          "text": "Який артикль для 'Buch'?",
          "options": [
            { "key": "a", "text": "der" },
            { "key": "b", "text": "die" },
            { "key": "c", "text": "das" }
          ],
          "correct": "c"
        }
      ]
    }
  ]
}
```
- **Відображення:** програвач квізу: показ питань по блоках, вибір варіанту, перевірка за `correct`. Реалізація форми може бути іншою (напр. один масив питань без blocks), якщо це відображається коректно і зберігається в `content` узгоджено.

---

### 6.8. scenario

- **Призначення:** інтерактивний сценарій з гілками вибору (Ф-004).
- **Поля форми:** редактор нод: для кожної ноди — id, type (`situation` | `choice` | `consequence` | `explanation`), text; для type `choice` — масив choices з text, nextNodeId, isCorrect, feedback (обов'язково для неправильних).
- **Вимоги:** мінімум 3 гілки в сценарії; у кожного choice є feedback (пояснення помилки).
- **Приклад content у БД:**
```json
{
  "nodes": [
    {
      "id": "start",
      "type": "situation",
      "text": "Ви в офісі. Колега просить допомогти з презентацією."
    },
    {
      "id": "n1",
      "type": "choice",
      "text": "Як ви відповідаєте?",
      "choices": [
        { "text": "Ja, gerne.", "nextNodeId": "ok", "isCorrect": true, "feedback": "" },
        { "text": "Nein.", "nextNodeId": "consequence1", "isCorrect": false, "feedback": "В офісі краще погодитися ввічливо. 'Ja, gerne.' — типова формула." }
      ]
    },
    {
      "id": "ok",
      "type": "consequence",
      "text": "Колега дякує. Ситуація вирішена."
    },
    {
      "id": "consequence1",
      "type": "explanation",
      "text": "Культурний контекст: у німецькому офісі очікується готовність до співпраці."
    }
  ]
}
```
- **Відображення:** Scenario Engine: показ поточної ноди, переходи по `nextNodeId`, при неправильному виборі — показ `feedback`. Схему нод можна розширювати (напр. додати `image_url`, `audio_url`), зберігаючи сумісність із поточною логікою на фронті.

## 7. Узагальнення щодо `content`

- **Один тип матеріалу = одна логіка форми і один спосіб відображення.** Структура JSON в `content` визначається цим типом і може відрізнятися від прикладів вище (інші ключі, вкладені об'єкти, додаткові поля).
- **Форма:** поля вводу налаштовуються під обраний тип; при збереженні вони збираються в один об'єкт `content` і відправляються в API. Можна використовувати різну реалізацію (проста форма, drag-and-drop, візуальний редактор нод для scenario тощо).
- **Відображення:** при читанні матеріалу з API фронт отримує `type` і `content` і рендерить відповідний компонент (VideoPlayer, TextBlock, VocabularyList, QuizPlayer, ScenarioEngine тощо). Як саме інтерпретувати ключі в `content` — вирішує фронтенд у межах узгодженої схеми для цього типу.
- **Еволюція:** нові типи матеріалів або зміни в структурі контенту (напр. додавання полів для аудіо, зображень) реалізуються через нові/оновлені схеми в `content` і відповідні форми та вьюери, без зміни типу колонки в БД.

## 8. Приклад екрану форми (псевдо-макет)

**Сторінка: Редагування курсу «Німецька A1»**

- Вкладки або секції: «Курс» | «Модулі» | «Матеріали».
- **Курс:** поля назви, опису, мови, категорії, чекбокс «Опублікований». Кнопка «Зберегти».
- **Модулі:** список модулів (назва, порядок). Кнопки «Додати модуль», «Редагувати», «Видалити». При виборі модуля — перехід до матеріалів цього модуля.
- **Матеріали (модуль «Модуль 1: Знайомство»):** список матеріалів (тип, назва, порядок). Кнопки «Додати матеріал», «Редагувати», «Видалити».
- **Модалка/сторінка «Додати матеріал»:** вибір типу (dropdown) → підвантаження полів контенту за типом (наприклад, для video — одне поле «YouTube ID»; для scenario — редактор нод). Спільні поля: назва матеріала, порядок. Збереження → POST/PATCH з `type`, `title`, `order_index`, `content`.

Такий підхід дозволяє мати одну гнучку форму з динамічними полями контенту за типом і однаково коректно зберігати та відображати різні типи матеріалів, використовуючи універсальне поле `content` в БД.
